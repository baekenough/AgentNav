# 06. 핵심 가설 및 측정 지표

---

## 한 줄 요약

`agents.json`이 있는 사이트에서 에이전트는 DOM 파싱 없이도 동일하거나 더 나은 작업을 수행할 수 있다는 가설을 정량적으로 검증한다.

---

## 7.1 핵심 가설

> **"사이트 소유자가 에이전트용 안내 파일을 제공하면,
> 에이전트가 DOM 파싱 대비 현저히 적은 토큰으로
> 동일하거나 더 나은 작업 수행을 할 수 있다."**

이 가설은 세 가지 전제를 포함한다:

1. **토큰 효율성**: 구조화된 안내 파일이 HTML/DOM 파싱보다 훨씬 적은 토큰을 소비한다.
2. **작업 완료율**: 안내 파일을 사용하는 에이전트가 시행착오 없이 작업을 더 높은 성공률로 완료한다.
3. **속도 개선**: 에이전트가 첫 번째 유의미한 액션에 도달하는 시간이 단축된다.

---

## 7.2 측정 지표

### Primary Metrics

| 지표 | 정의 | 단위 |
|------|------|------|
| **토큰 사용량 (Token Usage)** | 동일 작업 완료까지 소비된 총 토큰 수 | tokens |
| **작업 성공률 (Task Success Rate)** | 에이전트가 의도한 결과를 달성한 비율 | % |
| **첫 액션까지 시간 (TTFA)** | 에이전트가 첫 번째 유의미한 액션을 취하기까지 걸린 시간 | seconds |

### Secondary Metrics

| 지표 | 정의 | 단위 |
|------|------|------|
| **라운드트립 횟수** | 작업 완료까지 에이전트-서버 간 요청 횟수 | count |
| **오류 복구율** | 에이전트가 실패 후 올바른 경로로 전환한 비율 | % |
| **안내 파일 활용률** | agents.json을 먼저 탐색한 후 작업을 시작한 비율 | % |

---

## 7.3 측정 방법론

### 실험 설계

동일한 작업을 두 조건에서 실행하여 비교한다:

| 조건 | 설명 |
|------|------|
| **Control (Without)** | `agents.json` 없음. 에이전트가 DOM 파싱 또는 일반 웹 탐색으로 작업 수행 |
| **Treatment (With)** | `agents.json` 제공. 에이전트가 안내 파일을 먼저 읽고 작업 수행 |

### 토큰 사용량 측정

```
측정 범위:
  - 에이전트 초기 컨텍스트 ~ 작업 완료 확인까지의 전체 토큰
  - Input tokens + Output tokens 합산
  - Tool use tokens 포함

측정 도구:
  - Claude API: usage.input_tokens + usage.output_tokens
  - GPT API: usage.total_tokens
  - 로그 파일에 작업별로 기록
```

### 작업 성공률 측정

```
성공 기준 (작업별 사전 정의):
  - "블로그 글 목록 가져오기" → 올바른 API 엔드포인트로 데이터 반환
  - "사이트에서 할 수 있는 작업 나열" → agents.json의 actions 섹션 내용 정확히 반영
  - "특정 페이지 탐색" → 올바른 경로(URL)로 이동

판정 방법:
  - 자동: API 응답 데이터 유효성 검증
  - 수동: 에이전트 출력과 기대 결과 비교 (3인 독립 평가)
  - 동의율 < 90%인 경우 재평가
```

### TTFA (Time to First Action) 측정

```
정의:
  - 작업 지시 시점 ~ 에이전트의 첫 번째 외부 API/URL 호출 시점

측정 방법:
  - 에이전트 로그의 타임스탬프 파싱
  - "첫 번째 유의미한 액션" = DOM 파싱이 아닌 목적 지향적 API 호출

예외 처리:
  - 에이전트가 안내 파일 자체를 먼저 읽는 경우: agents.json 요청은 TTFA 시작으로 간주
```

### 실험 조건

```
샘플 사이트:
  - 정적 블로그 (GitHub Pages 기반)
  - 간단한 SaaS 랜딩 + API

테스트 작업:
  - T1: 콘텐츠 목록 조회
  - T2: 특정 항목 상세 정보 확인
  - T3: 사이트 기능 파악 및 요약

반복 횟수:
  - 조건당 각 작업 5회 반복 (총 30회 실험)
  - 동일 프롬프트, 동일 에이전트 설정
```

## 7.4 성공 기준

가설이 검증되었다고 판단하는 임계값:

| 지표 | 최소 기준 | 목표 기준 |
|------|-----------|-----------|
| 토큰 사용량 절감 | ≥ 50% 감소 | ≥ 80% 감소 |
| 작업 성공률 향상 | Control 대비 +10%p | Control 대비 +25%p |
| TTFA 단축 | ≥ 30% 단축 | ≥ 60% 단축 |

> **참고**: Vercel agent-browser 사례에서 DOM 대신 의미 정보만 전달 시 93% 토큰 절감, 도구 수 감소 시 성공률 80% → 100% 달성을 확인했다. 이 프로젝트의 목표 기준은 해당 선례를 참고한다.

### 가설 기각 조건

다음 중 하나라도 해당되면 가설은 현재 설계에서 기각된다:

- 토큰 절감이 30% 미만
- 작업 성공률이 동등하지 않거나 낮음
- TTFA가 오히려 증가 (에이전트가 agents.json 탐색 자체에서 막히는 경우)

기각 시에는 파일 포맷 또는 에이전트 발견 메커니즘을 재설계한다.
